## 1. 核心定位

HackThonGo 是一个面向单场黑客松活动的极简管理系统。

关键约束：
- 单赛事：一个系统实例只服务一场比赛，不支持在同一实例内创建多个比赛。
- 无参赛者账号：参赛者匿名提交，通过邮件接收结果。
- 多轮评审：支持初赛/复赛/决赛。
- 统一大池：晋级和排名永远基于同一全量项目池；赛区仅用于标签、筛选和统计。

---

## 2. 产品目标

### 2.1 业务目标
- 主办方在 1 天内完成赛事配置并可对外报名。
- 参赛者 8 分钟内完成一次性提交。
- 评审分配自动化覆盖 90% 以上任务。
- 轮次结束后 10 分钟内完成晋级名单确认与通知发送。

### 2.2 成功指标（KPI）
- 报名转化率 = 成功提交 / 报名页 UV。
- 评分完成率 = 已完成评分任务 / 已分配任务。
- 晋级时效 = 轮次截止到晋级名单发布的时间。
- 邮件送达成功率 = 成功发送 / 总发送。

## 3. 单赛事业务边界（必须遵守）

### 3.1 单赛事模式定义
- 系统初始化时自动生成且仅保留一个 `Competition`（或 `EventConfig`）记录。
- 管理员只能“编辑赛事配置”，不能“新建第二场比赛”。
- 前后端路由与 API 使用单赛事语义（`competition` 单数）。

### 3.2 赛区与晋级统一池规则
- 每个项目可选绑定一个 `region`（如北京/上海/深圳）。
- `region` 只用于：筛选、统计、展示、运营分组。
- `region` 不用于：独立排名、独立晋级池、赛区配额晋级。
- 每轮晋级计算范围是该轮全部有效项目（全局池）。

### 3.3 晋级计算口径
- 排名口径：按项目轮次平均分从高到低全局排序。
- 晋级规则（TOP_N / TOP_PERCENT / SCORE_THRESHOLD）作用于全局排序结果。
- 管理员可人工微调最终名单，但系统默认建议名单来自全局池。

### 3.4 不可违反的硬约束（Hard Invariants）
- HI-01：系统内仅存在一个可用赛事配置（单赛事）。
- HI-02：赛区是标签维度，不能作为晋级池切分条件。
- HI-03：晋级计算只能在轮次进入 `JUDGING_CLOSED` 后执行。
- HI-04：项目只有在“有效评分数 >= 最低评审数阈值”时才可进入自动晋级计算。
- HI-05：参赛提交不可编辑，评分提交默认不可回退（除系统管理员审计操作）。

## 4. 用户角色与权限

### 4.1 角色
- ADMIN：赛事运营与管理。
- JUDGE：只评审分配给自己的项目。
- PARTICIPANT：无账号，仅公开提交。

### 4.2 权限矩阵

## 5. 功能范围（MVP）

### 5.1 赛事配置（单实例唯一）
- 赛事名称、说明、时间范围、状态。
- 微信二维码（入群/小助手）配置。
- 赛区字典维护（可选，供项目选择标签）。
- 评分维度配置（如创新性、技术难度、完成度）。

### 5.2 动态表单
- 报名表（registrationForm）与项目提交表（submissionForm）。
- 字段类型：text/textarea/email/phone/url/select/multiselect/file/date。
- 支持必填、排序、校验规则、字段说明。

### 5.3 公开提交
- 无登录提交。
- 一次性提交，不可编辑。
- 生成唯一回执号。
- 发送确认邮件（失败可记录并重发）。

### 5.4 评委与任务分配
- 创建评委账号并启用。
- 自动分配（均衡随机 + 去重）。
- 手动调整（增删个别 assignment）。

### 5.5 评分
- 评委仅可见自己的任务和自己的评分。
- 按评分维度录入分数、评语。
- 状态：PENDING -> IN_PROGRESS -> COMPLETED。
- 只有维度分完整且提交成功的任务才计入“有效评分数”。

### 5.6 晋级
- 按轮次执行“全局池排名”。
- 系统按规则计算建议名单。
- 管理员确认/微调后批量写入晋级结果。
- 自动创建下一轮 ProjectRound 与任务分配。

### 5.7 通知与导出
- 报名确认邮件。
- 晋级/淘汰通知邮件（不含具体评分）。
- 导出项目、评分、晋级结果（CSV）。

## 6. 关键流程

### 6.1 管理员流程
1. 进入“赛事配置”（不是创建比赛）。
2. 配置动态表单、评分标准、轮次规则、赛区字典。
3. 创建评委账号。
4. 开放报名。
5. 报名截止后初始化首轮 ProjectRound。
6. 自动分配评审任务并手动微调。
7. 跟踪评分完成度。
8. 执行全局池晋级计算并确认名单。
9. 系统派发下一轮任务并发送通知。

### 6.2 参赛者流程
1. 访问公开提交页。
2. 填写基础信息 + 项目内容 + 可选赛区标签。
3. 提交后获得回执号。
4. 通过邮件接收结果通知。

### 6.3 评委流程
1. 登录评委端。
2. 查看被分配项目。
3. 对每个项目评分并提交。
4. 查看本人历史评分记录。

## 7. 详细需求（FR）

### 7.1 赛事配置
- FR-001：系统只允许一个赛事配置记录。
- FR-002：管理员可编辑赛事配置，但不可新增第二个赛事。
- FR-003：赛事状态支持 DRAFT/REGISTRATION/IN_PROGRESS/COMPLETED。

### 7.2 动态表单
- FR-010：支持字段增删改与排序。
- FR-011：前后端使用同一套校验规则。
- FR-012：项目提交时保存表单快照，避免配置变更污染历史数据。

### 7.3 提交
- FR-020：公开 API 支持匿名提交。
- FR-021：提交成功必须生成唯一回执号。
- FR-022：邮件失败不回滚提交，但必须记录失败原因。

### 7.4 分配
- FR-030：每个项目可配置分配 N 位评委。
- FR-031：自动分配应尽量均衡评委工作量。
- FR-032：同一评委在同一轮不得重复评同一项目。

### 7.5 评分
- FR-040：评分维度来源于赛事配置。
- FR-041：分值必须在合法范围内。
- FR-042：默认提交后不可修改（可配置截止前允许修改）。

### 7.6 晋级（统一池硬约束）
- FR-050：排名计算范围为当前轮次全部有效项目。
- FR-051：赛区字段不参与分池和晋级切分。
- FR-052：晋级规则只对全局排序结果生效。
- FR-053：管理员可微调名单，但系统建议名单必须来自全局池。
- FR-054：晋级后自动创建下一轮 ProjectRound 链路。
- FR-055：执行晋级前，轮次状态必须为 `JUDGING_CLOSED`。
- FR-056：项目必须达到“最低评审数阈值”（例如 3）才参与自动晋级排序。
- FR-057：未达到阈值的项目默认标记为 `INSUFFICIENT_REVIEWS`，需管理员补评或人工处理。

### 7.7 通知与导出
- FR-060：支持报名确认邮件模板。
- FR-061：支持晋级/淘汰通知模板。
- FR-062：支持导出项目/评分/晋级结果。

## 9. 排名与晋级算法定义（重点）

### 9.1 输入集
- 当前轮次所有 `ProjectRound`。
- 每个项目轮次下全部 `COMPLETED` 的 `JudgeAssignment.totalScore`。

### 9.2 计算步骤
1. 过滤出“有效项目”：有效评分数 >= 最低评审数阈值（配置项，默认 3）。
2. 计算项目平均分 `avgScore`（该轮次所有有效评分的均值）。
3. 对有效项目按 `avgScore desc` 全局排序。
4. 根据规则选出建议晋级名单：
- TOP_N：前 N 名。
- TOP_PERCENT：前 X%。
- SCORE_THRESHOLD：分数 >= 阈值。
5. 管理员可人工微调（但不允许按赛区切池截断）。
6. 写回 `promotionStatus` 并为晋级项目创建下一轮记录。
7. 未达评审阈值项目标记 `INSUFFICIENT_REVIEWS`，进入补评或人工决策队列。

### 9.3 同分处理（确保结果可复现）
- 一级：`avgScore` 高者优先。
- 二级：评分完成数更多者优先。
- 三级：提交时间更早者优先。
- 四级：`receiptNumber` 字典序。

### 9.4 赛区使用边界
- 允许在看板中按赛区查看排名分布。
- 不允许按赛区单独“截断”晋级名单。
- 不允许赛区配额直接覆盖全局晋级规则。

### 9.5 示例（验证“同一大池”）
- 输入：A/B/C 三个赛区共 100 个项目，规则为 `TOP_N = 20`。
- 计算：100 个项目统一排序，不分赛区子榜。
- 输出：前 20 名晋级；可能出现某赛区晋级多、某赛区晋级少，这是规则允许结果。

### 9.6 边界条件与处理策略
- EC-01（TOP_N）：若“有效项目数 < N”，系统不自动发布名单，提示“需补评或管理员降低 N 并留痕审批”。
- EC-02（TOP_PERCENT）：晋级人数按 `ceil(validCount * percent / 100)` 计算，至少 1（当 validCount > 0）。
- EC-03（SCORE_THRESHOLD）：可出现 0 个晋级项目，系统允许并要求管理员二次确认。
- EC-04（评分未完成）：`INSUFFICIENT_REVIEWS` 项目不得被系统自动晋级，但管理员可人工 override，必须写审计日志。
- EC-05（并列边界）：若最后一名存在并列，按 9.3 同分规则稳定决策，确保结果可复现。

## 11. API 设计（MVP）

### 11.1 公开 API
- `POST /api/projects`：匿名提交项目。

### 11.2 评委 API
- `GET /api/assignments`：获取我的任务。
- `POST /api/assignments/:id/scores`：提交评分。
- `GET /api/judge/profile`：评委资料。

### 11.3 管理员 API
- `GET /api/competition`：获取单赛事配置。
- `PUT /api/competition`：更新单赛事配置。
- `POST /api/rounds`：新增轮次。
- `POST /api/judges`：新增评委。
- `POST /api/assignments`：创建/调整分配。
- `PUT /api/project-rounds/:id/promotion`：单项目晋级决策。
- `POST /api/promotions/bulk`：批量晋级决策。
- `POST /api/projects/:id/receipt/resend`：重发确认邮件。

### 11.4 接口规范
- 400：参数错误（字段级错误）。
- 401/403：未认证或权限不足。
- 409：冲突（如重复分配、重复回执号）。
- 500：内部错误（附可追踪日志 ID）。

## 12. 技术架构

### 12.1 技术栈
- Next.js 14+（App Router）
- React 18 + TypeScript
- Tailwind CSS + shadcn/ui
- PostgreSQL + Prisma
- NextAuth.js（仅 ADMIN/JUDGE）
- Nodemailer（SMTP）

### 12.2 关键实现点
- 单赛事初始化脚本：首次启动自动插入默认赛事记录。
- 动态表单：配置驱动渲染 + Zod 动态校验。
- 分配算法：均衡随机，支持手动微调。
- 晋级引擎：严格全局池计算，不走赛区分池。
- 审计日志：分配、评分、晋级、邮件重发全记录。

## 13. 非功能需求

### 13.1 性能
- 100 项目/10 评委：列表与常规查询 < 1 秒。
- 1000 项目/50 评委：分配与晋级计算在可接受时间内完成。

### 13.2 可用性
- 移动端优先，关键操作控件 >= 48px 高度。
- 所有关键操作具备 loading/success/error 状态。
- 提交与评分具备幂等防重。

### 13.3 安全
- 输入校验、输出编码、防 XSS/SQL 注入。
- 基于角色的接口与页面权限控制。
- CSRF 防护（登录态写操作）。
- 上传文件白名单与大小限制。

### 13.4 可维护性
- 业务分层：`app` / `components` / `lib` / `types`。
- 关键算法（分配/晋级）必须有单元测试。
- 关键流程（提交/评分/晋级）有 E2E 冒烟测试。

## 15. 测试与验收

### 15.1 重点测试
- 单赛事约束：无法创建第二场比赛。
- 赛区标签场景：有/无赛区都走同一晋级池。
- 三种晋级规则计算正确。
- 评分不足阈值时，不得自动进入晋级名单。
- 自动分配 + 手动调整后无重复任务。
- 多评委并发评分一致性。
- SMTP 失败记录与重发。
- 移动端提交流程。

### 15.2 验收标准
- 管理员可在“单赛事配置”完成全流程准备。
- 参赛者匿名提交成功并收到回执邮件。
- 评委仅可见本人任务并提交评分。
- 晋级名单由全局池计算，赛区不切分晋级池。
- 通知与导出可稳定执行。

## 16. 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| 将赛区误用为分池 | 晋级口径混乱 | 在产品、接口、测试中固化“全局池”规则 |
| 自动分配不均衡 | 评审延期 | 可视化负载 + 手动微调入口 |
| 邮件送达不稳 | 参赛者体验下降 | 重试队列 + 管理端重发 |
| 表单配置错误 | 报名失败率上升 | 发布前预览与校验 |
| 高并发评分冲突 | 数据不一致 | 事务与幂等键控制 |

## 18. 术语

- 全局池：某轮次全部有效项目集合。
- 赛区标签：项目归属信息，仅用于筛选与统计。
- ProjectRound：项目在某一轮的状态记录（评分、排名、晋级）。
- AdvancementRule：晋级规则（TOP_N/TOP_PERCENT/SCORE_THRESHOLD）。
